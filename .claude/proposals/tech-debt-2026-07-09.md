## Arquitectura — resuelto en esta pasada (no estaba en el reporte original)

Encontrado por 2 agentes de exploración durante el plan de upgrade de Next,
ver plan en `.claude/plans/adicional-a-esos-ajustes-mighty-catmull.md`:

1. **Alto — bug real de prod:** `lib/api/endpoints/transactions.ts` hacía
   `fetch()`/`<a href>` directo a `BASE_URL` (backend real) saltando
   `/api/proxy`, rompía en Docker (host inalcanzable desde browser) y
   bypaseaba la inyección de `Authorization` del proxy. **Resuelto**:
   reescrito para pasar por el proxy; el proxy ahora soporta passthrough
   binario (uploads multipart, descargas xlsx) — verificado end-to-end
   contra un backend real en Docker (register → workspace → descarga
   template → upload dry-run, bytes intactos).
2. **Medio — bug activo confirmado (currency ≠ COP en prod):**
   `PaymentModal.tsx` hardcodeaba `'COP'` porque `ScheduleTable.tsx` nunca le
   pasaba el prop `currency` que sí tenía disponible. **Resuelto**: prop
   agregado y forwarded correctamente.
3. **Medio:** 7 páginas (no 5 — el scan inicial no vio
   `debts/[debtId]/page.tsx` ni `savings/[goalId]/page.tsx`) duplicaban
   `workspacesApi.get(workspaceId)` cuando el layout ya tenía la lista
   completa vía `workspacesApi.list()`. **Resuelto** sin Context nuevo:
   las 7 páginas ahora llaman `.list()` + `.find()`, y el memoization
   automático de `fetch()` de Next.js dedupea las llamadas idénticas
   dentro del mismo request — verificado con logs de backend real: 1 sola
   `GET /workspaces` por render en vez de 2.
4. **Bajo:** `types/api.ts` tenía una `interface ApiError` muerta (0
   importadores) con el mismo nombre que la `class ApiError` real de
   `client.ts`, riesgo de confusión futura. **Resuelto**, borrada.
5. **Medio, confirmado con el usuario (backend rota refresh tokens
   single-use):** race condition en `client.ts` — 401s concurrentes
   disparaban múltiples refresh en paralelo, el backend invalida el token
   del perdedor → logout espurio. **Resuelto**: mutex compartido
   (`refreshPromise`), cubierto por test (`client.test.ts`).
6. Cookie names/opciones (`access_token`, `refresh_token`, httpOnly,
   sameSite, maxAge) estaban hardcodeadas en 3 archivos sin type-safety
   entre ellos. **Resuelto**: extraídas a `lib/auth/constants.ts`.
7. `src/app/api/auth/token/route.ts` era dead code confirmado (0
   referencias). **Borrado**.

## Next.js 14 → 15.5.18

Migración chica (nadie usaba `next/image`, `useFormState`, ni el patrón
`HydrationBoundary` que el propio CLAUDE.md documenta pero no existe en el
código real). `await cookies()` en 5 archivos, `eslint-config-next` bumped.

**Hallazgo no anticipado (no aplicado — decisión consciente):**
`NEXT_PUBLIC_API_URL` se hornea en build time por Next.js en TODO código
compilado por webpack — incluye los route handlers server-side
(`/api/proxy`, `/api/auth/refresh`), no solo el bundle de cliente.
Confirmado con repro limpia (build sin Railway) que sin `ARG`/`ENV` en el
builder stage el valor queda `undefined` y cae al fallback
`host.docker.internal`, rompiendo el proxy en cualquier build fuera de
Railway. Se agregó el `ARG` al Dockerfile y luego se revirtió a pedido del
usuario: Railway ya inyecta la variable como build-arg automáticamente y
prod funciona hoy sin el `ARG` explícito. Queda como debt conocido, no
bloqueante mientras el único target de deploy sea Railway — reconsiderar si
se agrega otro entorno de build (CI que construya la imagen, otro PaaS,
etc.), porque ahí sí rompería silenciosamente (Railway lo
compensaba con inyección automática de build-args). **Resuelto**: `ARG
NEXT_PUBLIC_API_URL` + `ENV` agregado al Dockerfile.
